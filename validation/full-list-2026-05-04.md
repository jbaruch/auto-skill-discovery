## POC-affecting findings

### 1. Meta-target prevalence is the dominant non-filter finding (~60–65 companies)

Companies whose own product is itself an agent / skill / MCP / eval / observability / agent-runtime platform. Discovery on these will produce findings tightly coupled to the POC's own surface, raising "this skill is just wrapping their MCP" risk. Roughly 25% of `RUN_DISCOVERY` is in this shape.

**Already publishes a public skills/ asset:** Black Forest Labs (`skills/` repo following an `agentskills.io` spec — first-class prior art for this POC's format), Bright Data (`brightdata/skills`), Arize (`arize-skills`), Telnyx (`telnyx-toolkit`).

**Ships an official MCP server:** Granola, Braintrust, Cursor, Supabase, GitHub, Postman, Monday.com (`mondaycom/mcp`), ElevenLabs (`elevenlabs-mcp`), RunPod, Canva (`Canva Dev MCP` for Cursor and Claude Code), Vercel (AI SDK + MCP), Pomerium, deepset (Hayhooks).

**Product is itself an agent or agent-runtime platform:** Modal, Trigger.dev, Unblocked, Ona (Gitpod rebrand), n8n, Hugging Face, Lovable, Superagentic AI, Aily, Namespace Labs, Hacktron AI, Sixfold, Ufonia, Waypoint AI, Jointly AI, Zeta Labs, Legora, Workera (Sage), tldraw (AI agent starter kits), Cognee (= AI Memory Engine — already publishes Claude Code plugin), Liquid AI (LFM foundation models), Orbit (AI-native dev env), Iterate (Slack AI agent), TESSA / Altrina (BPO agent), HITL / humanintheloop-relay.com.

**Eval / observability platform:** Langfuse, Comet (Opik), Arize (Phoenix), Snorkel AI (Toolathlon, OpenEnv, terminal-bench), Encord (Active), Raindrop, Braintrust.

**MCP / agent infrastructure or protocol-level work:** Alpic (MCP deploy), RAW Labs (MXCP/Aimbient), Always Further (agent kernel sandbox), Cerebras, Together AI (Harbor), EXO Labs (43.8k★ on `exo`), Featherless.ai, Forter (`trusted-agentic-commerce-protocol`), Faculty (`a2a-extension-object-schemas`), incident.io ("AI SRE"), Qodo (Agent Skills), Databricks (managed MCP), DOTSAFE (decentralized-identity).

**Adjacent dev-tools / OSS handbooks:** PostHog (full handbook OSS), Spotify (Backstage 21k+★), Linear (already-validated MCP), Intercom (Fin agent), Manychat (chatbot), PyannoteAI (voice infra).

**Implication for POC:** the spec's `agentic_landscape.competition_assessment` field handles per-company assessment, but with this base rate the POC needs a primary differentiation framework. The vanilla "wrap-the-API" `api_wrapper` target kind is rarely the right play — `workflow_skill` or `domain_skill` recipes layered on top of existing MCPs are the higher-leverage shape.

### 5. Name-normalization issues — UNKNOWN verification protocol added

The dedup script handles whitespace / case but doesn't catch:

- Generic concepts misread as entities: "HITL" — resolves via privacy-policy scan
- Marketing taglines misread as company names: "Purpose AI at Every Scale" — resolves via tagline rule
- Names with multiple unrelated referents: Orbit / TESSA / Iterate / rumble — resolve via qualified-AI search; ORYX, Bis, DOTSAFE, LDC Media — resolve via deeper protocol
- Vanity domains pointing to thin / non-software entities: toyo-shikisai.com — Step 1 fetch
- Possible typos or zero web hits: Asparanta — actually a real Dutch ICT consultancy (just thin); LIAVELLA — Shopify leather-bags brand
- Entities outside software altogether: EXONA Laboratories Pakistan (medical lab)

**Implication for filter:** UNKNOWN bucket needs the five-step verification protocol before bucketing — agents lazy-bucketing UNKNOWN was the dominant failure mode in the first pass. The protocol cuts the bucket from 6.1% to 0.3% of the list. Now codified in `SKILL.md` Step 2.

### 6. Government / community MCP servers — discovery should include both

Government / public-sector entries already have community-built MCPs even where no official one exists.

- Regeringskansliet (Swedish government) → community-built `isakskogstad/Riksdag-Regering-MCP`

**Implication for POC:** discovery's `agentic_landscape` should scan community-built MCPs alongside official ones, or a generated skill may duplicate existing community work.

## Reconciliation against the four-bucket filter

The filter's four-bucket design holds. Three refinements warranted by evidence and now codified in the skill:

1. **`MEGA_CORP` active hidden-multi-brand check.** Many parents (IKEA, Caesar Groep, Reshift, Monterro, Eon Digital, Bis) don't read as multi-brand from the name alone — count distinct GitHub orgs / docs portals / brand pages before defaulting to RUN_DISCOVERY.
2. **`SELF_OR_NA` widened to "not an engineering organization".** Explicitly includes VCs, podcasts, schools, branding agencies, newsletters, trade pubs, non-tech consumer brands, research consortia, professional bodies, stealth / empty-stub sites, personal coaching brands, non-software entities.
3. **`UNKNOWN` five-step verification protocol.** Mandatory before bucketing — direct domain fetch, literal-name search + privacy-policy scan, qualified-AI search, tagline / DBA expansion, acronym expansion. Cuts the bucket from 6.1% to 0.3%.

The previously-collapsed `PRODUCT_COMPANY` vs `NEEDS_DISCOVERY` split remains correctly collapsed; routing-bucket variance is too high to enumerate priors and discovery's BUILD/SKIP machinery handles fine-grained classification on real evidence.

`UNKNOWN` proved its value in the residue case (ORYX SOFTWARE) — the bucket should remain even if it is rarely populated after the protocol.

## Implications for the broader POC (not filter-specific)

Beyond filter refinements, three findings shift the POC's strategy:

1. **Default differentiation hypothesis should be "scenario-driven recipe on top of the existing MCP," not "wrap the API."** ~25% of routing candidates already publish official or community MCPs / skills / agent platforms. The `api_wrapper` target kind in the discovery output contract is structurally the wrong answer for that population.
2. **Discovery's `agentic_landscape` section should explicitly distinguish official-MCP, community-MCP, and existing-skill-on-Tessl-or-elsewhere.** The competition assessment is finer-grained than the contract currently captures.
3. **Black Forest Labs' `skills/` repo following the `agentskills.io` spec is direct prior art** worth examining before this POC settles its own skill format — both for format alignment and as a possible reference target.

## Sample size and caveats

Validated: 311 unique entities (the 312-row deduplicated list contained one entry — `BCG Platinion` — treated as the same firm as Boston Consulting Group, and HITL + Chishingo Ventures t/a HITL resolved to the same entity, but both rows are bucketed RUN_DISCOVERY for filter accounting). Evidence per company: between 1 and 6 web fetches per row in the first pass; second pass added the five-step verification protocol on the 19 UNKNOWNs (~3–10 fetches per row).

The lowest-confidence entry remaining is the single ORYX SOFTWARE residue. RUN_DISCOVERY routing was not deeply audited against discovery output — a future validation pass running discovery on a stratified sample of RUN_DISCOVERY routes would test whether the filter's high routing rate (79.5%) translates to high BUILD-verdict rate or to high SKIP-rate.
